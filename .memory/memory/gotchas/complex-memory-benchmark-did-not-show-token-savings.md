The May 20, 2026 Memory token-savings benchmarks do not support using Memory as a token-saving claim. The original complex pilot completed 12 implementation runs across p-limit, zod, and vite with gpt-5.5 low/high and found Memory at +27.5% net uncached tokens and +15.7% gross tokens versus control, with +1.0 average blind quality. A later curated rerun first created reusable repo Memory once per target repo, then completed 16 ready-subset runs; curated Memory still used more tokens: +105,934 net uncached tokens (+21.7%) and +1,642,830 gross tokens (+25.9%) overall, and +64,606 net (+15.7%) / +1,446,622 gross (+26.7%) on complex pairs. Curated Memory also added +133.9s total implementation time and +1,056.6s setup/curation/load overhead, while all hidden verifiers and focused checks passed in both conditions. Blind quality was slightly better with curated Memory (+0.375 overall, +0.5 complex). Do not use these runs as public proof of token savings; the stronger hypothesis is that Memory may trade extra context and setup cost for better or more consistent patch quality.
