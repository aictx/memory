As of 2026-06-10 a full rework of Memory is being planned (proposed, not yet decided): drop the per-task load/save choreography in favor of a graphify-style loop — one-time agent-driven indexing, on-demand query, diff-driven sync. The graph should model the product layer (features, status, decisions, open questions) anchored to code surfaces (routes, commands, jobs, paths), not code structure itself. Motivations: the May 20, 2026 benchmarks showed no token savings from per-task loading (see gotcha.complex-memory-benchmark-did-not-show-token-savings); the agent-facing surface is too large (23 CLI commands, 12 memory kinds) for reliable agent use; reference project github.com/safishamsi/graphify (~65k stars, YC S26) validates index-once/query-on-demand for the code layer and leaves the product layer unserved. Target persona under this pivot: developers running many projects with coding agents who lose product state between sessions. This challenges the broad-persona stance in synthesis.product-intent.
